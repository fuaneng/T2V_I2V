# InternVideo2 图文搜视频 Demo 审查与 V2 重新设计方案

## 1. 原代码存在的问题审查 (Code Review)

经过仔细审阅您的第一版演示代码，发现以下几个**致命的逻辑脱节**和**功能性缺失**，导致原系统无法真正跑通“文搜视频”和“图搜视频”：

1. **多模态特征提取函数缺失 (`model_wrapper.py`)**
   - **问题**：`streamlit_ui.py` 中调用了 `model_engine.encode_text()` 和 `model_engine.encode_image()`，但 `model_wrapper.py` 里根本没有这两个方法，只有 `encode_video`。
   - **后果**：前端点击搜索必然抛出 `AttributeError`。
2. **前后端与提取脚本的数据配置“割裂”**
   - **问题 1（数据库不同步）**：`extract_and_index.py` 建库写入了 `./internvideo_local_vms.db`，而 `vector_db.py`（供前端读库）指向了 `./internvideo_vms.db`，两者读写的文件不一致。
   - **问题 2（特征维度不同步）**：`extract_and_index.py` 创建集合时使用了 1024 维，而 `vector_db.py` 初始化使用的是 768 维。
   - **后果**：特征插入和检索在架构底层处于完全脱节的状态，会导致 Milvus 极大的兼容异常并找不到任何有效数据。
3. **`extract_and_index.py` 的插入逻辑松散**
   - **问题**：`extract_and_index.py` 内部自己调用了 `client.insert`，没有重用 `vector_db.py` 中定义好的管理器。
   - **后果**：代码冗余，扩展性差。

---

## 2. V2 重构方案与逻辑设计

为了彻底解决上述卡点，将项目重构并升级至 V2。

### 核心架构改造

1. **引入统一的配置中心 `config.py`**：将 `MODEL_PATH`, `DB_FILE_PATH`, `DIMENSION` (统一规定为 768) 抽离出来。不论是前端 UI、模型还是数据提取脚本，都从同一入口获取参数源，从根本上解决“数据库不一致”、“维度不一致”的问题。
2. **补全新模态的编码能力 (`model_wrapper.py`)**：专门开发 `encode_text` 和 `encode_image` 接口封装。我为您搭建了兼容这两种模态的接口标准。当您切回真实的 InternVideo2 模型实战时，只需将真实的推断代码无缝嵌入这个函数内即可（当前为符合维度的随机向量占位供链路跑通）。
3. **聚合与增强 Milvus 接口类 (`vector_db.py`)**：把数据库建库、清空重置、批量插入（`insert_batch`）等能力统一收归该类。所有脚本均通过 `MilvusManager` 读写，不再直接越权接触 pymilvus 底层 API。
4. **提升 Streamlit UI 的健壮性**：增加友好的查询状态过渡、捕获模型异常防止崩溃白屏、在侧边栏实时监测数据库体量状态。
